# Контракт приложения: бэкенд платформы вместо своего сервера

Этот документ — **источник правды** о том, как пользовательское приложение общается с
ProjectsFlow вместо собственного backend-процесса. Он описывает **реально реализованную**
поверхность (`server/src/presentation/app-runtime/appRuntimeRouter.ts`), а не намерения
дизайна: предыдущая спека обещала JWT, тогда как код использует серверные сессии, и
конвертация по спеке сломалась бы.

> Расхождение документа с кодом ловится тестом
> `server/src/presentation/app-runtime/appRuntimeRouter.contract.test.ts`: он сверяет
> список маршрутов ниже с фактическими роутами роутера. Меняешь роуты — правь и таблицы здесь.

## 1. Когда это применимо

Платформа **не запускает** пользовательские процессы. Она:

1. раздаёт **статическую сборку** на `<slug>.projectsflow.ru`;
2. на том же поддомене обслуживает `/api/auth/*` и `/api/data/*` силами App Runtime.

Значит приложение с собственным `server.js`, Express/Fastify и своей MySQL/Postgres в
платформе не заработает **никогда** — не из-за лимитов, а потому что исполнять его негде.
Чтобы такой проект ожил, его данные и авторизацию переводят на контракт ниже, а свой
сервер и драйвер БД удаляют.

Признаки проекта, которому нужна конверсия: в `package.json` есть `start`/`dev`, поднимающий
сервер; в зависимостях драйвер БД (`mysql2`, `pg`, `mongodb`); в коде свой HTTP-роутинг.

## 2. Модель

```
браузер → https://<slug>.projectsflow.ru/…        → статические файлы приложения
браузер → https://<slug>.projectsflow.ru/api/data → App Runtime → SQLite этого проекта
браузер → https://<slug>.projectsflow.ru/api/auth → App Runtime → пользователи приложения
```

**Проект определяется поддоменом**, а не телом запроса: host-middleware резолвит slug и
кладёт `projectId` в запрос до попадания в роутер. Клиент подделать его не может, поэтому
никаких `projectId`/`appId` в запросах приложения нет и быть не должно.

У каждого приложения **свои конечные пользователи** — они не связаны с аккаунтами
ProjectsFlow. Владелец проекта не является пользователем приложения.

## 3. Схема данных

Объявляется декларативно и хранится в `app_backends.schema_json`. Для App Runtime это
**белый список**: имена таблиц и полей никогда не берутся из запроса клиента, только отсюда.
Поэтому произвольный SQL и произвольные имена колонок недоступны by design.

```jsonc
{
  "tables": [
    {
      "name": "orders",
      "fields": [
        { "name": "customer", "type": "text", "required": true },
        { "name": "phone",    "type": "text", "sensitive": "pii" },
        { "name": "total",    "type": "real" },
        { "name": "paid",     "type": "bool" }
      ],
      "rules": { "read": "owner", "write": "authenticated", "delete": "owner" }
    }
  ]
}
```

**Типы полей:** `text` · `int` · `real` · `bool` · `datetime`.
Модификаторы: `required`, `unique`, `sensitive: 'secret' | 'pii'`.

**Служебные колонки ведёт рантайм — не объявляй их:**
`id`, `owner_id`, `created_at`, `updated_at`.
`owner_id` заполняется текущим пользователем и является основой правила `owner`.
`updated_at` — версия строки для оптимистичной конкуренции.

**Правила доступа** (`rules`) на таблицу:

| Значение | Кто может |
|---|---|
| `anyone` | без авторизации |
| `authenticated` | любой вошедший пользователь приложения |
| `owner` | только владелец строки (`owner_id` = текущий пользователь) |

Обязательны `read` и `write`. Опционально `create` / `update` / `delete` уточняют write
пооперационно; если правило не указано — действует `write`.

## 4. Аутентификация

Токен — **серверная сессия**, а не JWT. Приложение получает его при входе и шлёт в
заголовке `Authorization: Bearer <token>`. Проверка на сервере, отозвать можно.

| Метод | Путь | Назначение |
|---|---|---|
| POST | `/api/auth/signup` | регистрация по email + паролю |
| POST | `/api/auth/signin` | вход, возвращает `{ user, token }` |
| POST | `/api/auth/signout` | завершить сессию |
| GET | `/api/auth/me` | текущий пользователь по Bearer-токену |
| GET | `/api/auth/config` | какие способы входа включены у приложения |
| GET | `/api/auth/google/start` | начать OAuth-вход через Google |
| GET | `/api/auth/google/callback` | колбэк Google |

`AppUser` = `{ id, email }`. Пароли хранит платформа — своей таблицы пользователей
приложению заводить не нужно.

## 5. Data-API

Все пути относительно поддомена приложения.

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/data/:table` | выборка |
| POST | `/api/data/:table` | создать строку |
| PATCH | `/api/data/:table/:id` | изменить строку |
| DELETE | `/api/data/:table/:id` | удалить строку |
| POST | `/api/data/:table/bulk` | создать пачкой |
| PUT | `/api/data/:table/bulk` | заменить пачкой |
| POST | `/api/data/:table/update-many` | массовое изменение по фильтру |
| POST | `/api/data/:table/:id/restore` | восстановить мягко удалённую строку |

**Параметры выборки.** Зарезервированы `sort`, `dir` (`asc`\|`desc`), `limit`, `offset`.
**Любой другой query-параметр трактуется как фильтр равенства по полю** — имена сверяются
со схемой, неизвестные отклоняются.

```
GET /api/data/orders?paid=false&sort=created_at&dir=desc&limit=20
```

Тело bulk-запросов — либо голый массив, либо `{ "items": [...] }`.

## 6. Ошибки

| HTTP | `error` | Когда |
|---|---|---|
| 400 | `bad_request` | таблица/поле вне схемы, схема невалидна |
| 401 | `auth_failed` | нет или недействителен Bearer-токен |
| 403 | `access_denied` | правило таблицы не пускает |
| 404 | `not_provisioned` | у проекта не активирован бэкенд |
| 409 | `user_exists` | регистрация на занятый email |
| 409 | `duplicate_value` | нарушен `unique` |
| 413 | `storage_full` | исчерпана квота хранилища проекта |

`duplicate_value` намеренно не называет колонку: иначе ответ становится оракулом,
по которому можно перебирать содержимое чужой таблицы.

## 7. Как конвертировать существующее приложение

Пример на приложении вида «Node-сервер + SQL» (`server.js`, `src/backend/*-store.mjs`,
статика в `public/`).

**Шаг 1. Разделить.** Фронт (`public/`) остаётся и станет артефактом. Серверная часть
подлежит удалению, но сначала из неё вынимается знание: какие сущности и поля есть.

**Шаг 2. Объявить схему.** Каждая SQL-таблица → таблица в `schema_json`. Типы свести к пяти
поддерживаемым. Колонки `id`/`created_at`/владелец — выбросить, их ведёт рантайм.

**Шаг 3. Переписать обращения к данным.** Было:

```js
// src/backend/mariadb-store.mjs
const [rows] = await pool.query('SELECT * FROM orders WHERE paid = ? ORDER BY created_at DESC', [0]);
```

Стало (в браузере, своего сервера больше нет):

```js
const res = await fetch('/api/data/orders?paid=false&sort=created_at&dir=desc', {
  headers: { Authorization: `Bearer ${token}` },
});
const rows = await res.json();
```

**Шаг 4. Переписать авторизацию** на `/api/auth/*`, удалить свою таблицу пользователей и
хеширование паролей.

**Шаг 5. Удалить** `server.js`, драйвер БД из зависимостей и серверные скрипты из
`package.json`. Оставить сборку статики, если она есть.

**Шаг 6. Опубликовать** собранную статику как артефакт сайта — тем же путём, что и обычный
статический проект.

### Что честно не переносится

- **Серверная логика между запросами** (крон, вебхуки, очереди, интеграции с внешними API
  по секрету) — App Runtime это CRUD над таблицами, а не среда исполнения. Если приложение
  без такого не живёт, его место не здесь.
- **Произвольный SQL** — джойны, агрегаты, транзакции на несколько таблиц. Доступны фильтр
  равенства, сортировка и пагинация.
- **Существующие данные.** Контракт описывает форму, а не миграцию строк. Перенос
  содержимого старой БД — отдельная работа.

Эти ограничения лучше проговорить владельцу проекта **до** конверсии, а не после того, как
у него удалили рабочий сервер.
