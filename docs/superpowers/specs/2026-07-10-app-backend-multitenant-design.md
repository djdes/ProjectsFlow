# Дизайн: мультитенантный backend для сгенерированных приложений (SQLite-на-проект)

**Дата:** 2026-07-10
**Статус:** черновик на ревью (владелец)
**Автор:** тимлид-сессия (Claude)

> Продолжение self-serve воркер-раннера. Пока конвейер отдаёт **статику**
> (`<slug>.projectsflow.ru`). Этот спек — как дать сгенерированным проектам
> **бэкенд** (вход, пользователи, база, правила) без сторонних сервисов, дёшево
> по ресурсам, с лимитом 100 МБ на проект.

---

## 1. Цель и не-цели

**Цель.** Сгенерированное приложение может иметь: регистрацию/вход своих
пользователей, собственную базу данных, правила доступа к данным — и всё это:
- **без сторонних сервисов** (свой код, не Supabase/Firebase);
- **без отдельного сервера/БД на проект** (иначе сотни проектов = сотни живых
  процессов, тяжело);
- **с квотой 100 МБ на базу проекта**;
- **под ключ** — воркер настраивает всё сам, у пользователя ноль ручных шагов.

**Не-цели (пока).**
- Произвольный серверный код на проект (уникальные функции) — Фаза 3.
- Файловое хранилище/загрузки, realtime, OAuth-вход — Фаза 2.
- Замена основного backend'а ProjectsFlow — это отдельная подсистема для
  ПОЛЬЗОВАТЕЛЬСКИХ приложений, не для самой платформы.

## 2. Архитектура (суть)

Три вещи, живут в разных местах:

| Слой | Где собирается | Где работает/лежит |
|---|---|---|
| Фронтенд приложения (SPA) | GitHub Actions | у нас, статикой на `<slug>.projectsflow.ru` |
| **App Runtime** (общий бэкенд) | — | у нас, **одна программа на все проекты** |
| База проекта (SQLite-файл ≤100 МБ) | — | у нас на диске, `apps-data/<project_id>.sqlite` |

Поток запроса:
```
конечный юзер → фронт (<slug>.projectsflow.ru) → <slug>.projectsflow.ru/api/* → App Runtime
             → открывает файл-базу проекта → проверяет правила/квоту → ответ
```

**Ключевая идея:** проект — это **лёгкий файл на диске + строка в реестре**, а не
запущенный процесс. Простаивающий проект ест ноль CPU/RAM. Одна программа-рантайм
обслуживает тысячи проектов; нагрузка растёт с числом ЗАПРОСОВ, а не проектов.

*(«SQLite на тенанта» — проверенный паттерн: Cloudflare D1, Turso, Fly.io.)*

## 3. Компоненты

1. **App Runtime** — Node/Express-модуль у нас: auth, data-API, проверка правил и
   квоты, изоляция по проекту. Stateless, кроме открытия нужного файла-базы.
2. **Per-project SQLite** — файл на проект: системные таблицы (`_users`,
   `_sessions`) + таблицы приложения + метаданные.
3. **App Registry** — таблица в основной БД ProjectsFlow: `project_id`, статус,
   объявленная схема, `usage_bytes`, `storage_limit_bytes` (по умолчанию 100 МБ),
   секрет app-ключа (хеш).
4. **Provisioning** — юзкейс: создать файл-базу + применить схему + сгенерить ключ
   при пометке проекта «app с бэкендом».
5. **JS-SDK** (`@projectsflow/app-client`) — тонкий клиент для фронта: `signUp`,
   `signIn`, `from('table').select/insert/update/delete`. Воркер вставляет его.

## 4. Модель данных

**В основной БД (миграция `db/NNN_app_backend.sql`):**
```
app_backends(
  project_id CHAR(36) PK,
  status ENUM('none','active') DEFAULT 'none',
  schema_json MEDIUMTEXT,           -- объявленные таблицы + правила
  app_key_hash VARCHAR(255),        -- секрет для фронта (храним хеш)
  usage_bytes BIGINT DEFAULT 0,
  storage_limit_bytes BIGINT DEFAULT 104857600,  -- 100 МБ
  created_at, updated_at
)
```

**Внутри файла-базы проекта (SQLite):**
- `_users(id, email, password_hash, created_at, ...)` — энд-юзеры ЭТОГО приложения
  (отдельные от аккаунтов ProjectsFlow).
- `_sessions(token_hash, user_id, expires_at)` — сессии.
- Таблицы приложения — из `schema_json` (напр. `posts`, `comments`).
- `_meta(key, value)` — версия схемы и пр.

## 5. Авторизация приложения

- У каждого приложения **свои пользователи**. Регистрация/вход по **email + пароль**
  (bcrypt/argon2), выдаём **JWT** (подписан секретом приложения).
- Endpoints: `POST /api/auth/signup`, `/signin`, `/signout`, `GET /api/auth/me`.
- Токен хранится на фронте; каждый data-запрос несёт его → рантайм знает
  `current_user` для правил доступа.
- OAuth (Google и т.п.) — Фаза 2.

## 6. Data-API

- Универсальный REST/query поверх таблиц приложения:
  `GET /api/data/:table?filter&sort&limit`, `POST/PATCH/DELETE /api/data/:table`.
- **MVP:** CRUD + фильтры по полям + сортировка + пагинация. *(Связи/агрегации —
  позже; см. решение 4.)*
- **Правила доступа на таблицу** (аналог RLS), объявлены в `schema_json`, напр.:
  `posts: { read: 'anyone', write: 'owner' }` — рантайм проверяет ПЕРЕД операцией.
- Только **параметризованные** запросы (жёсткая защита от SQL-инъекций); имена
  таблиц/полей — из белого списка схемы, не из сырого ввода.

## 7. Квота 100 МБ (решение 3: жёсткий стоп)

- Перед каждой записью — быстрая проверка размера файла (`usage_bytes`, обновляем
  после операций записи).
- `usage < 100 МБ` → пишем. `≥ 100 МБ` → **HTTP 413 «storage_full»**, чтение
  продолжает работать.
- Владельцу в UI ProjectsFlow: «Хранилище: X / 100 МБ» + предупреждение на 80%.
- Лимит на проект настраиваемый (`storage_limit_bytes`) — задел под тарифы.

## 8. Изоляция и безопасность

- Проект определяется по **поддомену** запроса (`<slug>` → `project_id`) —
  фронт не может подделать чужой проект.
- Каждый запрос открывает **только** файл своего проекта; путь строится из
  валидированного `project_id`, не из ввода (нет path-traversal).
- App-ключ (для серверных операций/провижининга) — секрет, храним хеш.
- Правила доступа проверяются **на сервере**; фронту не доверяем.
- Пароли энд-юзеров — сильный хеш; JWT-секрет уникален на приложение.

## 9. Как воркер настраивает (self-serve, под ключ)

1. Проект помечается «app с бэкендом» (тип проекта / тумблер).
2. Воркер при сборке приложения **объявляет схему** (таблицы + правила) — кладёт
   `projectsflow.app.json` в репо ИЛИ через MCP-tool `pf_declare_app_schema`.
3. Provisioning: создаём файл-базу + применяем схему + генерим app-ключ +
   `app_backends.status='active'`.
4. Воркер генерит фронт с нашим SDK (endpoint = `/api`, ключ из env при сборке).
5. Фронт собирается на GitHub → публикуется на `<slug>.projectsflow.ru`.
6. Готово: приложение с логином и данными живое.

## 10. Встраивание в ProjectsFlow (Clean Arch)

- `domain/app-backend/` — AppBackend, схема, правила, ошибки (0 внешних зависимостей).
- `application/app-backend/` — порты (`AppDatabaseStore`, `AppBackendRepository`) +
  юзкейсы (`ProvisionAppBackend`, `RunAppQuery`, `AuthenticateAppUser`, `CheckQuota`).
- `infrastructure/` — `SqliteAppDatabaseStore` (открытие/пул файлов, применение
  схемы, замер размера), `DrizzleAppBackendRepository`.
- `presentation/app-runtime/` — роуты `/api/auth/*`, `/api/data/*`; монтируются в
  host-middleware для board/app-поддоменов (там, где сейчас отдаётся статика).
- Реестр приложений — миграция в основной БД (см. §4).
- Клиент: новый пакет `client-sdk/` (или отдельный npm-пакет `@projectsflow/app-client`).

## 11. Этапы

- **MVP** — App Runtime + SQLite-на-проект + auth(email) + CRUD data-API +
  правила доступа + квота 100 МБ + provisioning + SDK. → приложения с логином и
  данными работают.
- **Фаза 2** — файловое хранилище (загрузки, свой лимит), OAuth-вход, realtime
  (SSE/WebSocket), богаче data-API (связи, агрегации).
- **Фаза 3** — произвольная серверная логика на проект: serverless-функции
  (изолированный рантайм, scale-to-zero) для приложений, которым нужен свой код.

## 12. Принятые решения (дефолты — поменять на ревью)

1. **Хранилище: SQLite-файл на проект** (рекоменд.). Плюсы: тривиальный кап по
   размеру файла, ноль процессов на проект, бэкап/удаление = операция с файлом.
   Минус: один писатель на файл (для небольших приложений — ок).
   *Альтернатива:* общий Postgres со схемой-на-проект (лучше конкурентность, но
   кап и бэкап сложнее).
2. **API на том же поддомене:** `<slug>.projectsflow.ru/api/*` (same-origin, без
   CORS и куки-проблем). Host-middleware маршрутизирует `/api` в App Runtime.
3. **Квота — жёсткий стоп** на запись при 100 МБ (предсказуемо, как free-tier
   везде) + предупреждение на 80%.
4. **Data-API в MVP — CRUD + фильтры/сортировка/пагинация** (без связей/агрегаций
   в первой версии).

## 13. Риски / честно

- Это **разовая, но серьёзная** разработка — по сути свой мини-BaaS: auth + data +
  правила + квота, где **безопасность критична** (чужие данные, инъекции, изоляция).
  → MVP делаем узким, с тестами на каждый слой, безопасность ревьюим отдельно.
- SQLite-конкурентность: для приложений с высокой нагрузкой на запись один файл —
  узкое место. Для массы небольших приложений — не проблема; для «выросших» —
  путь миграции на Postgres-схему (решение 1 обратимо на уровне порта `AppDatabaseStore`).
- Провижининг схемы от воркера: воркер должен генерить корректный
  `projectsflow.app.json`; нужен строгий валидатор схемы на нашей стороне.
