# Единое пространство, мгновенное делегирование и Telegram-меню «по ответственным»

Дата: 2026-07-13. Статус: дизайн утверждён, ждёт плана реализации.

## 1. Задача

Три связанные части (формулировка заказчика — в скобках):

1. **Telegram-меню задач по ответственным** («когда вызываешь — чтобы предлагал глянуть
   таски по ответственным, с возможностью завершить, комментировать и перейти в задачу,
   чтобы срок или ответственного сменить; с указанием сроков»).
2. **Приглашения — с per-project на единое пространство** («убрать по проектам приглашения
   и сделать только в одно пространство, где есть куча проектов и куча других ответственных»).
3. **Делегирование без принятия/отказа** («у тасков, когда делегируется кому-то, — без
   возможности принятия или отказа»); перепроверить логику во всех местах: чат,
   уведомления, «делегировано» и т.д.

Части связаны: сегодня делегирование не-участнику проекта работает через статус
`pending_invite` (accept = вступление в проект). В едином пространстве все участники —
«свои», поэтому этот флоу выпиливается целиком (подтверждено заказчиком).

## 2. Утверждённые решения

| Вопрос | Решение |
| --- | --- |
| Что такое «вызов» бота | `/tasks` в личке И «пустое» @упоминание в группе (упоминание с текстом по-прежнему создаёт задачу) |
| Охват списка по ответственным | Всё пространство — и в личке, и в группе |
| Доступ приглашённого в пространство | Ко ВСЕМ проектам пространства, включая будущие; личные «Входящие» каждого остаются приватными |
| Миграция существующих участников | Открыть всё: участники проектов становятся участниками пространств и видят все проекты в них |
| Invite-delegate флоу («пригласить в проект при делегировании») | Удаляется полностью — не понадобится |
| Архитектура членства | Вариант A: `workspace_members` — единственный источник доступа, читается «насквозь» через реализацию `ProjectMemberRepository` (без фан-аута и синков) |
| Смена срока/ответственного из TG | Только переход в веб по deep-link (inline-пикеры в TG — out of scope) |

## 3. Часть «Единое пространство»

### 3.1 Данные

- `workspace_members.role`: `ENUM('owner','member')` → `ENUM('owner','editor','viewer')`.
  Существующие `member` → `editor`.
- Новая таблица `workspace_invites` — зеркало `project_invites`: `id`, `workspace_id`,
  `role ENUM('editor','viewer')`, `token CHAR(64) UNIQUE`, `email NULL` (информационное),
  `expires_at` (TTL 7 дней), `accepted_at`, `accepted_by_user_id`, `created_by_user_id`,
  `created_at`.
- `project_invites` замораживается: новые не создаются; непринятые токены продолжают
  работать — accept зачисляет в **пространство** проекта (код-адаптация, без миграции токенов).
- `project_members` перестаёт быть источником доступа. Таблица остаётся ТОЛЬКО как
  хранилище персональных пер-проектных настроек (`notification_prefs`, `sort_order`,
  `is_favorite`, `favorite_sort_order`); строки создаются лениво при первой записи настроек.
- `projects.owner_id` остаётся кешем владельца (используется дайджестами/диспетчером), не трогаем.

### 3.2 Доступ (сервер)

- Интерфейс `ProjectMemberRepository` НЕ меняется. Переписывается Drizzle-реализация:
  - `findForProject(projectId, userId)` → join `projects.workspace_id` → `workspace_members`;
    роль маппится 1:1 (owner→owner, editor→editor, viewer→viewer).
  - **Инвариант приватности Входящих**: для проектов `is_inbox=true` доступ имеет только
    владелец, независимо от пространства (спец-ветка в реализации). Делегаты видят
    отдельные inbox-задачи через существующий `taskAuthorization` (creator или delegate).
  - `listByProject` → участники пространства проекта (для inbox — только владелец).
  - `countOwners` → owners пространства (защита «последнего owner» переезжает на ws-уровень).
  - `listSharedUsers(userId)` → участники общих ПРОСТРАНСТВ (пул для пикеров/меншенов).
  - `listProjectsForUser` → проекты всех пространств, где юзер участник, + его inbox.
- `requireProjectAccess`, матрица `permissions.ts` и все ~30 use-case'ов не меняются.
  Admin-bypass сохраняется.
- Use-cases приглашений: `CreateWorkspaceInvite`, `AcceptWorkspaceInvite`,
  `GetInviteByToken` (различает оба типа токенов), `ListWorkspaceInvites`,
  `DeleteWorkspaceInvite` — адаптация project-аналогов. Project-level инвайт use-cases
  и роуты `POST/GET/DELETE /api/projects/:id/invites` удаляются.
- `project_join_requests` (вход по git-коллизии) остаются; resolve зачисляет в пространство
  проекта с ролью editor.
- Управление командой переезжает на уровень пространства: смена роли, удаление участника,
  передача владения — операции над `workspace_members` (расширение `WorkspaceService`).
  Прямое добавление по email (`addMember`) остаётся как второй способ рядом с токен-ссылкой.
- `HubMembershipSync` удаляется: членство default-хаба управляется приглашениями напрямую,
  а не деривится из project_members.
- `WorkspaceService.moveProject`: копирование участников больше не нужно — перенос проекта
  в другое пространство просто меняет его аудиторию на аудиторию целевого пространства
  (задокументированное следствие модели).
- Семантика видимости: доступ = проекты с `workspace_id = W` для каждого W, где юзер
  участник. Default-хаб как ВЬЮХА по-прежнему агрегирует все проекты по участию
  (включая проекты чужих пространств) — это отображение, не доступ.

### 3.3 Клиент

- `InviteDialog` — приглашение в пространство (роль editor/viewer, email опционален,
  токен-ссылка); вызов из настроек пространства и из проекта (ведёт в то же место).
- `InvitePage` (`/invite/:token`) принимает оба типа токенов (превью: название
  пространства/проекта, приглашающий).
- `TeamSection` на странице проекта показывает участников пространства (read-only,
  ссылка «Управлять» → настройки пространства). `WorkspaceSettingsPage` — единая
  страница команды.
- Пикеры ответственных, @-меншены, `NotifyAudienceControl`, «Из знакомых» — данные
  приходят из тех же API (`listMembers`/`listSharedMembers`), сервер теперь отдаёт
  ws-участников; клиент почти не меняется.

## 4. Часть «Делегирование без принятия/отказа»

- Все пути создания делегации (`CreateTask`, `DelegateExistingTask`,
  `ReassignTaskDelegation`, Telegram-composer) создают её сразу `status='accepted'`
  (`responded_at = created_at`). Особая ветка isSelf в `DelegateExistingTask` становится
  общим случаем.
- Задача мгновенно видна делегату: блок «Поручено мне», merge в доску «Входящих»
  (`listAcceptedDelegatedTo`), чекбокс «выполнено» доступен сразу (`canModify` по
  `accepted` выполняется тривиально).
- Уведомление «вам поручена задача» (`task_delegation`) остаётся во всех каналах
  (in-app + email + TG), но БЕЗ кнопок Принять/Отказать; в TG к карточке цепляются
  «✅ Завершить / 💬 Комментировать / Открыть».
- **Удаляются**: `AcceptTaskDelegation`, `DeclineTaskDelegation`; REST
  `POST /api/delegations/:id/accept|/decline` и `GET /pending`; клиентские
  `handleAcceptDelegation/handleDeclineDelegation`; TG-callbacks `da:`/`dd:` и кнопки;
  уведомление «X принял задачу» (ветка resolution:'accepted'); `InviteAndDelegateTask`,
  статус `pending_invite`, `revert_to_user_id`-логика, `POST /:taskId/invite-delegate`,
  UI invite-delegate (drag на не-участника, `PendingCard`, кнопка «Вступить»);
  янтарный бейдж «ждёт ответа» и счётчик «N ждёт ответа» (`AssignedToMeBlock`,
  `DelegationBadge`).
- `task_delegation` уходит из actionable-наборов в ОБОИХ синхронных местах
  (`DrizzleNotificationRepository.countActionableUnread` и
  `GetActivityFeed.ACTIONABLE_TYPES`) — бейдж «Действие» больше не считает делегации.
- **Остаются**: `WithdrawTaskDelegation` (создатель забирает) и
  `RelinquishTaskDelegation` (делегат снимает с себя — право отказа постфактум).
  На relinquish создателю приходит уведомление «X снял(а) с себя задачу» + email
  (переиспользуем механизм `task_delegation_resolved` с resolution `declined`,
  тексты/письмо переформулируются).
- Требование «делегат — участник» сохраняется, но означает участие в пространстве
  (через переписанный репозиторий).
- Telegram-composer: сейчас задача `+Проект текст @Делегат` создаётся во «Входящих»
  создателя и переносится в проект (`MoveTaskToProject`) только после accept делегата —
  эта развилка существовала из-за неопределённости согласия. Теперь задача с названным
  проектом создаётся сразу в проекте с уже принятой делегацией; ветка «перенос при
  accept» удаляется вместе с `handleAccept`.
- ENUM статусов в БД не сужаем: `pending/declined/pending_invite` остаются как
  историческое значение, новые строки с ними не создаются.

## 5. Часть «Telegram: вызов → задачи по ответственным»

- Триггеры меню: `/tasks` в личке; в группе — @упоминание бота без другого текста
  (упоминание с текстом — как сейчас, создание задачи через composer; привязка группы —
  `telegram_group_owners`).
- Экран 1 «По ответственным» (охват — все проекты пространств вызывающего; в группе —
  пространство владельца привязки): кнопки `👤 Имя (N)` по каждому ответственному
  с открытыми задачами + `Без ответственного (N)` (без активной делегации = за создателем).
  Внизу вторичная кнопка `📁 По проектам` — существующая навигация `bt:p:` сохраняется.
  Новый callback-префикс `ba:<userId>` / `ba:none`.
- Экран 2 «Задачи ответственного»: карточки задач (лимит 12, сортировка: просроченные →
  по сроку → без срока). Карточка: **название задачи** (сейчас `/tasks` подписывает кнопки
  обрывком описания — чиним), проект, `⏰ срок` через `formatDeadlineRu` с пометкой
  просрочки. Кнопки: «✅ Завершить» (`nd:`, edit-message «✅ Завершено» + «Отменить»),
  «💬 Комментировать» (`nc:`, force-reply), url-кнопка «Открыть в ProjectsFlow»
  (deep-link `?task=` — в вебе меняются срок и ответственный). Reply на карточку =
  комментарий (существующий механизм `telegram_task_messages`).
- Все действия гейтятся членством вызывающего (тот же `findForProject`) и проталкивают
  SSE-события в веб.

## 6. Уведомления и чат — сводка изменений

- Новый тип `workspace_invite` (payload: workspaceId, workspaceName, role, token,
  inviter) + ветка в `NotificationItem` с кнопкой «Принять» + email-шаблон
  (адаптация `inviteEmail`). Тип `project_invite` остаётся рендериться для старых записей.
- Actionable-наборы: `{workspace_invite, project_invite, join_request}`;
  `task_delegation` исключается.
- `task_delegation_resolved`: остаётся только для «снял с себя» (см. §4).
- Попутный фикс найденного рассинхрона: `chat_mention` создаётся сервером, но
  отсутствует в клиентском union и `NotificationItem` (рендерится пустышкой) —
  добавить тип и рендер.
- Чат: хаб-чат уже workspace-level; после удаления `HubMembershipSync` его участники =
  участники пространства (приглашённые). `parseChatMentions` уже работает по
  `WorkspaceMember` — без изменений.
- Дайджесты/EOD: получатели и `members.length > 1` теперь считаются по ws-участникам
  через тот же репозиторий; группировка по делегату не меняется.

## 7. Миграции БД (append-only, MariaDB-синтаксис; последняя — db/109, целимся в 110–112, перепроверить номера при реализации)

1. `workspace_members.role`: расширить ENUM (`'owner','member','editor','viewer'`) →
   `UPDATE member→editor` → сузить ENUM (`'owner','editor','viewer'`).
2. `CREATE TABLE workspace_invites` (см. §3.1).
3. Бэкфилл членства: для каждого не-inbox проекта P (workspace W) каждый участник из
   `project_members(P)` → `INSERT IGNORE workspace_members(W)`; роль — высшая из его
   проектных ролей в проектах W (owner проекта, не являющийся owner_user_id пространства,
   получает editor; «последний owner» пространства не понижается).
4. `task_delegations`: `pending`/`pending_invite` → `accepted`,
   `responded_at = COALESCE(responded_at, NOW())`.

Клиент и сервер катятся одним деплоем (push в main → автодеплой + `npm run db:migrate`
на сервере в пайплайне).

## 8. Проверка

- `npm run typecheck`, `npm run lint`, `npm run build`; серверные интеграционные точки —
  ультракод-ревью (многоагентная проверка всех затронутых мест: чат, уведомления,
  «делегировано», TG, дайджесты, EOD, пикеры, меншены) перед коммитом.
- После деплоя — прод-верификация демо-аккаунтами по отработанной схеме: создать
  пространство → инвайт-ссылка → принять вторым demo-аккаунтом → видит все проекты →
  делегирование (мгновенно у делегата, без кнопок принятия) → relinquish → уведомление
  создателю → зачистка demo-аккаунтов. TG-меню (`/tasks`, «пустое» упоминание в группе)
  проверяет владелец вручную — бот боевой.

## 9. Out of scope

- Inline-смена срока/ответственного прямо в Telegram (пикеры дат/людей) — только deep-link в веб.
- BOD-дайджест («с чего начать») — отдельная незавершённая фича, не трогаем.
- Выборочный доступ к отдельным проектам внутри пространства.
- Мульти-tenant биллинг/лимиты на уровне пространств.
