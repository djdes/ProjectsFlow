## 2. Реализация

### 2.1 Визуал / вёрстка (п.1)

- Не ломать существующую загрузку/сохранение (репозитории, поля). Это рефактор представления.

### 2.2 Визуальное разделение блоков (п.3)
- Каждая автоматизация — **отдельная карточка/панель** с заголовком, краткой подписью и тонким
  разделителем между. Группировка (предлагаемая): (1) Диспетчер (мастер + критерии + лимит + пауза),
  (2) Мультизадачный воркер, (3) Автор коммитов + флаги (CLAUDE.md, UltraCode) + деплой,
  (4) Telegram-группа, (5) Ежедневная сводка, (6) Авто-обработка по коммитам. Чёткая визуальная
  сегментация, чтобы не сливалось.

### 2.3 Логика мастер-переключателя (п.2)
- **Определить набор «автоматизаций», которыми управляет мастер.** Рекомендация (уточнить у
  пользователя в начале — оставить комментарий-обоснование в коде): мастер агрегирует **основные
  автоматизации**: диспетчер (`enabled`), мультизадачный воркер, ежедневная сводка (`digest.enabled`),
  авто-обработка по коммитам (`commitSyncEnabled`). (Флаги типа ignoreClaudeMd/UltraCode — это
  модификаторы, НЕ самостоятельные автоматизации → в агрегат не входят.)
- **Правила:**
  - `master = OR(всех включённых автоматизаций)` — производное, не отдельное хранимое булево
    (или синхронизируемое).
  - Включена хотя бы одна → мастер показывается ВКЛ.
  - Все выключены → мастер ВЫКЛ.
  - **Пользователь выключает мастер → выключить ВСЕ** (батч-апдейт всех автоматизаций в off,
    сохранить через соответствующие репозитории/эндпоинты).
  - **Пользователь включает мастер** — поведение УТОЧНИТЬ у пользователя. Дефолт-предложение:
    включить только диспетчер (минимально, `enabled=true`), не трогая остальные; ИЛИ восстановить
    последнее ненулевое состояние (если хранить snapshot). Выбрать с пользователем, задокументировать.
- Реализовать аккуратно: оптимистичные апдейты + сохранение; не словить рассинхрон между
  `automation` и `digest` (разные репозитории/эндпоинты — батч может быть в 2 запроса; обработать
  ошибки/частичный успех).

### 2.4 История group-id + имя группы (п.4)
- **История ID групп.** Показывать ранее введённые `telegram_group_chat_id`. Источник — distinct
  значения по проектам текущего пользователя из `project_digest_settings`.
  - Новый GET-эндпоинт **в существующем `server/src/presentation/digest/routes.ts`** (НЕ трогать
    `http.ts`!), напр. `GET /api/me/telegram-groups` → `[{ chatId:number, title:string|null }]`
    (или `GET /api/projects/:id/telegram-groups` с тем же смыслом). Запрос: distinct
    `(telegram_group_chat_id, telegram_group_title)` где chat_id не null и проект принадлежит юзеру.
  - UI: заменить «голый» text-инпут на **combobox / `<datalist>`** с подсказками (id + title), плюс
    ручной ввод сохранить. Выбор подсказки подставляет chat_id (+ title).
- **Резолв названия группы.** При вводе/выборе chat_id — попытаться **спарсить название** через
  Telegram `getChat` (бот) на сервере: новый эндпоинт в digest-роутере, напр.
  `POST /api/projects/:id/telegram-group/resolve` (body `{ chatId }`) → `{ title }`. Реализация —
  через `TelegramClient.getChat(chatId)`. Сохранять `title` в `telegram_group_title`.
  - **Мягкий фоллбэк:** если бот не в группе / нет прав / ошибка — вернуть `{ title:null }`, в UI
    показать только id (без падений, без блокировки сохранения).
- Миграция: вероятно НЕ нужна (история строится запросом из существующей таблицы). Если решишь
  кэшировать историю отдельно — новый файл `db/0NN_*.sql` (`ls db/ | sort`, номер ОТЛИЧНЫЙ от P2:
  договорённость — P3 берёт max+2). Лучше обойтись без миграции (distinct-запрос).

## 5. Edge-кейсы / риски

- Бот не состоит в группе → `getChat` ошибка → фоллбэк title=null, не блокировать сохранение.
- chat_id отрицательный/целый — валидация (как сейчас); пустое поле допустимо (без группы).
- Мастер-агрегат через 2 репозитория (automation + digest) → возможен частичный успех при сети;
  откатывать оптимистичный стейт при ошибке, показывать toast.
- Не задеть пер-проектную логику диспетчера/лимитов (только представление + агрегирующая логика).
- Distinct-история по проектам ПОЛЬЗОВАТЕЛЯ (не всех) — проверить скоуп прав в запросе.
