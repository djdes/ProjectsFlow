# Режим плана для задач (Plan → одобрение → исполнение) — Дизайн

**Дата:** 2026-06-04
**Статус:** утверждён, к реализации.

## Контекст
Раньше «домысливание» делал лёгкий триаж/readiness. Нужен полноценный план «как plan mode»:
по кнопке — code-aware планировщик исследует репозиторий, пишет план в комментарий,
человек одобряет/правит через привычные каналы (Telegram/дашборд), и только потом воркер
исполняет — уже по готовому плану, быстрее и без лишних вопросов. Если код успел измениться —
воркер сверяет план и адаптирует его перед исполнением.

## Решения (из брейншторма)
- Триггер: кнопка «🗺 Составить план» на карточке (ручной).
- Поверхность ревью: план = комментарий; approve/edit/вопросы — через существующий F11
  (Telegram-кнопки / reply / панель дашборда). Гейт воркера — через F11 (clarify-blocked).
- «Изменить» = перегенерация плана с текстовыми правками (итеративно).
- Планировщик — code-aware `claude -p` в репо (Read/Grep/Glob), без записи кода и без MCP;
  только возвращает JSON-план. Все REST-действия делает ralph.
- Триаж по умолчанию ВЫКЛ (`PF_TRIAGE` default off) — план заменяет его.

## Поток
1. **Кнопка (сайт):** постит комментарий через существующий `createComment`:
   `🗺 Запрошен план реализации` + `<!-- ralph-plan-request {"v":1} -->`. Новый write-эндпоинт не нужен.
2. **Детект (сервер):** `GET /agent/plan-requests?limit=N` (agent, Bearer) — задачи диспетчера,
   у которых есть `ralph-plan-request` и ещё нет `ralph-plan`. Зеркало `pending-ai-prompt-jobs`,
   но запрос по `task_comments` (маркер) + join projects (dispatcher_user_id = caller). Только чтение.
3. **Планировщик (ralph):** `Try-PlanTick` поллит эндпоинт (throttle), найдя запрос —
   паркует задачу (`awaiting_clarification`) и спавнит **детач** план-воркер (как ai-job-worker).
   План-воркер: `claude -p` в `repoPath` с `--allowedTools Read,Grep,Glob` + `plan-prompt.md`
   (вход: описание + комментарии + KB-дайджест). Возвращает JSON
   `{goal, steps[], criteria[], risks[], solutions[], assumptions[], files[], questions[]}`.
4. **Публикация плана (ralph):** постит комментарий с красивыми MD-блоками
   (🎯 Цель / 📋 Шаги / ✅ Критерии / ⚠️ Риски / 🛠 Решения / Допущения) + маркер
   `<!-- ralph-plan {"v":1,"baseSha":"<HEAD на момент плана>","files":[...]} -->`.
   Затем постит F11-вопрос `ralph-question` «Применить план?» с опциями `✅ Применить` /
   `✏️ Изменить` (allowOpen). Регистрация в F11 (`Register-PendingFromTaskComments`) →
   задача автоматически clarify-blocked (воркер не берёт) — ГЕЙТ переиспользует F11, без нового стейта.
5. **Ревью (F11):** вопрос уходит в Telegram + дашборд.
   - `Применить` → `ralph-answer` → задача разблокируется → воркер берёт её.
   - `Изменить`+текст → ответ распознаётся как правка плана (по id вопроса с префиксом `plan-`):
     ralph **сразу** перепаркует задачу (`awaiting_clarification`) и перезапустит планировщик
     с текстом правок → новый план + новый вопрос (петля). Перепарковка до того, как воркер
     схватит задачу (REST-move синхронно при детекте ответа).
6. **Исполнение (воркер) со сверкой свежести:** диспетчер подставляет последний `ralph-plan`
   в промпт (`{{APPROVED_PLAN}}` + `baseSha` + `files`). Воркер ПЕРЕД работой:
   - `git diff <baseSha>..HEAD -- <files>` (и/или перечитать файлы);
   - изменений нет → выполнять строго по плану (без переплана/лишних вопросов);
   - файлы изменились → перечитать текущий код, **скорректировать план** под актуальное
     состояние (убрать конфликты), кратко отметить правки комментарием, затем выполнять.
   Readiness-gate для задач с одобренным планом пропускается; сверка свежести — всегда.

## Изменения по репозиториям
- **ProjectsFlow:**
  - client: кнопка «🗺 Составить план» в карточке задачи (TaskDrawer) → `createComment` с маркером.
  - server: use-case `ListPlanRequests` + route `GET /agent/plan-requests` + wiring (http.ts/index.ts).
  - **БД-миграции НЕ требуются** (фича на маркерах-комментариях). Автодеплой прогонит `migrate.mjs` как no-op.
- **ralph:**
  - `prompts/plan-prompt.md` (UTF-8 BOM).
  - `dispatch.ps1`: `Get-PendingPlanRequests`, `Try-PlanTick`, спавн план-воркера (детач),
    распознавание ответа на plan-вопрос (Применить/Изменить) в F11-обработке, подстановка
    `{{APPROVED_PLAN}}` в Start-Parallel-Worker, skip readiness при наличии плана,
    триаж default-off, конфиг (`PF_PLAN`, `planModel`=opus, `planWatchdogSeconds`=180).
  - `ai-job-worker.ps1`: режим `plan` (code-aware spawn + публикация плана/вопроса по REST),
    либо отдельный `plan-worker.ps1`.
  - `worker-prompt.md`: шаг «Проверка актуальности плана» + плейсхолдер `{{APPROVED_PLAN}}`.

## Риски
- Гонка worker↔re-plan при «Изменить» → синхронная перепарковка при детекте ответа.
- Дрейф кода между планом и исполнением → сверка `baseSha..HEAD` в воркере (раздел 6).
- Стоимость планировщика (полный прогон в репо) → раз на задачу, gated `PF_PLAN`, модель в конфиге.
- Детект plan-requests по маркеру → серверный запрос с фильтром (как has_marker), троттлинг тика.
