Цель — превратить 8 коротких пунктов в понятные фичи с описанием сути, текущего
состояния, scope'а и открытых вопросов, сгруппированные по приоритету.

**Важное открытие при разведке:** значительная часть пунктов 5 и 6 — это НЕ greenfield.
На ветке `feat/agent-runner-loop` уже есть:
- таблица `agent_jobs` (очередь задач для агента) + миграции 014/015;
- модель «kanban agent runner» на базе `/loop` (Plan B v2) — Claude Code локально
  раз в N минут опрашивает очередь, забирает задачу, делает её в git-worktree, открывает PR;

То есть «передать мою автоматизацию» (п.5) и «чекбокс автоулучшения» (п.6) — это

**Решения, подтверждённые юзером:**

- П.6 — системный промпт автоулучшения хранится **и как флаг в БД (UI), и как .md-файл в репо**.
- П.7 — пока только задокументировать идею, без реализации сейчас.

# P0 — Быстрые UX-победы

## Фича №2. Счётчик комментариев на карточке задачи

**Суть простыми словами.** На карточке задачи (и в kanban, и в списке) должно быть
видно, сколько у неё комментариев — как сейчас видно число коммитов и вложений.

**Как сейчас.** Карточка ([KanbanCard.tsx:112-132](../../../client/src/presentation/components/tasks/KanbanCard.tsx#L112-L132))
показывает бейджи `commitCount` и `attachmentCount`. Комментарии **не показываются** —
их видно только открыв задачу. В доменной модели `Task`
([Task.ts](../../../client/src/domain/task/Task.ts)) есть `commitCount?` и `attachmentCount?`,
но **нет `commentCount`**. В списочном endpoint'е комментарии вообще не считаются.

**Что делаем.**
1. Бэкенд: в выдачу списка задач добавить агрегат `commentCount` (COUNT по `task_comments`,
   аналогично тому, как уже считается `commitCount`). Один JOIN/подзапрос.
2. Домен (client): добавить `commentCount?: number` в тип `Task`.
3. UI: бейдж с иконкой комментария рядом с существующими, в обоих представлениях
   (`KanbanCard.tsx` и `TaskListView.tsx`). Скрывать, если `0` (как сделано для коммитов).
4. MCP: добавить `commentCount` в выдачу `pf_list_tasks` (агенту полезно знать,
   что по задаче уже есть обсуждение).

## Фича №3. Порядок выполненных задач + настройка сортировки

**Суть простыми словами.** Сейчас выполненные задачи в колонке «Done» идут снизу-вверх
(старые сверху). Хочется, чтобы свежевыполненные были сверху. И/или — дать юзеру
переключатель порядка сортировки.

**Как сейчас.** В колонке сортировка строго по `position` ascending
([KanbanBoard.tsx:66-71](../../../client/src/presentation/components/tasks/KanbanBoard.tsx#L66-L71)).
Никакой пользовательской настройки нет; единственная сохраняемая преференция — тема
(localStorage, ключ `pf-theme`).

**Что делаем.**
1. **Дефолт:** в колонке `done` сортировать **descending** (свежие сверху). Остальные
   колонки — без изменений, там ручной порядок по `position` имеет смысл.
2. **Настройка:** переключатель «новые сверху / старые сверху», хранить per-user в
   localStorage (паттерн темы уже есть); миграция в БД — позже, если попросят синк.
3. UI переключателя — над доской или в меню колонки done.

## Фича №4. Доска задач — представление по умолчанию при входе в проект

**Суть простыми словами.** Заходишь в проект → сразу видишь kanban-доску, а не страницу
с описанием проекта.

**Решение:** Вариант A — доска становится «домом» проекта на `/projects/:projectId`,
обзор переезжает на `/projects/:projectId/overview`, на доску добавляется явный вход «Обзор».

# P1 — Функциональность

## Фича №1. Полный CRUD проектов через MCP + автосоздание git-репозитория

**Суть простыми словами.** Сейчас агент через MCP может читать проекты и задачи, создавать
задачи/комменты/credentials/KB, но **не может создать сам проект**. Нужно довести MCP до
полного управления: создать проект, отредактировать, заархивировать — и при создании
**автоматически заводить привязанный GitHub-репозиторий**.

**Что делаем.**
1. **MCP-tool'ы по проектам:** `pf_create_project {name, description?, createGitRepo?, gitRepoUrl?}`,
   `pf_update_project`, `pf_archive_project`.
2. **Автосоздание git-репо:** use-case `CreateProjectRepo` через GitHub API.
3. **Дочистить CRUD задач:** `pf_update_task` (title/description), `pf_delete_task`.
4. **Права:** мутации проверяют роль через `can(role, action)`.
5. README mcp-server обновить.

**Открытые вопросы (ответить перед исполнением):**
- Аккаунт/организация и токен для автосоздания репо на GitHub (это и есть неясная фраза
  про «креды в actions не передаём» — надо определить источник прав).
- `pf_archive_project` vs hard delete — подтвердить, что нужен только архив.
- Нужен ли агенту `pf_get_project` (get-by-id).

## Фича №8. Нормальная мобильная версия

**Суть простыми словами.** Сделать платформу удобной с телефона — в первую очередь kanban.

**Как сейчас.** Каркас (`AppShell`) адаптивен (сайдбар → drawer на <768px). Проблема —
kanban: колонки фикс-ширины `w-72`, горизонтальный скролл, на телефоне влезает ~1 колонка.

**Что делаем.**
1. **Kanban на мобиле:** табы по статусам (Backlog/Todo/In Progress/Done) — одна колонка на экран.
2. Адаптивная ширина колонок (`w-72` → `w-[85vw] sm:w-72`).
3. Пройтись по остальным экранам на 360–414px, починить переполнения.
4. Тач-таргеты ≥ 44px. На мобиле — «переместить в статус» через меню вместо drag.

`pf_list_pending_agent_jobs` уже возвращает pending-задачи по всем проектам юзера с

Не хватает «обёртки запуска» (твой `C:\www\ralph\dispatch.ps1`).

**Что делать в будущем:**

2. Обобщить `dispatch.ps1` в версионируемый шаблон `scripts/agent-dispatch.ps1.example`

## Фича №6. Чекбокс автоулучшения («логика Ярика»)

Галочка «авто-улучшение» у проекта: когда включена и очередь пуста — агент сам генерирует
задачи улучшения, реализует их с дебагом и проверкой, открывает PR.

**Что делать в будущем (решено: и БД-флаг, и файл):**
1. БД: `projects.agent_config` (JSON, минимум `autoImproveEnabled: bool`), галочка в UI.
   Файл в репо: `.projectsflow/auto-improve.md` — текст промпта, версионируется.
   MCP-tool `pf_get_project_agent_config`.
2. Поведение `/loop`: очередь пуста + флаг включён → режим генерации задач улучшения
   (`pf_create_task` с `delegated_to_agent`), далее обычный цикл реализации.
3. **Защита от runaway:** обязательный дневной/сессионный кап на автогенерацию.

**Открытые вопросы:** число капа; нужно ли подтверждение человеком (по умолчанию автономно,
PR всегда draft); дефолтное содержимое `.projectsflow/auto-improve.md`.

## Фича №7. Скилл чтения переписки (Telegram) → автосоздание задач

Скилл читает входящие ТГ-сообщения и превращает их в задачи на доске.

**Статус:** только зафиксировать идею.

**Черновик подхода.** Уже подключён `telegram` MCP-плагин. Но у Telegram Bot API **нет
истории/поиска** — бот видит только входящие в реальном времени. MVP: скилл ловит входящие,
выделяет «похоже на задачу», предлагает создать через `pf_create_task`. Серверный бот —
альтернатива на будущее. Открытые вопросы: маппинг чат → проект; подтверждающий шаг;
дедупликация.

## Verification (как проверять при исполнении)

Каждая фича проверяется отдельно; общий принцип — `npm run typecheck` + `npm run lint`
после фронтовых изменений, ручная проверка в `npm run dev`.

- **№4:** клик по проекту → сразу доска; обзор доступен по `/overview`.

- **№3:** завершить задачи в разное время → свежие сверху; переключатель меняет направление,
  переживает перезагрузку (localStorage).
- **№8:** DevTools на 375px → kanban читаем (табы по статусам), нет горизонтального скролла-ловушки.
- **№1:** через MCP `pf_create_project {createGitRepo:true}` → проект в UI, `gitRepoUrl` заполнен,
  репо создан на GitHub; `pf_update_task`/`pf_delete_task` работают.
- **№5, №6, №7:** не реализуются сейчас — проверка не требуется.
