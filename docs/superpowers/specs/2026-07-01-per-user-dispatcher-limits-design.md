# Пер-юзерные лимиты диспетчера (attribution → инициатор)

> Дизайн одобрен владельцем 2026-07-01. Строим поверх уже смердженной фичи usage-limits
> (`db/082–084`): ledger, окна 5ч/7д, тарифы, блок-на-claim (402), UI. Меняем **атрибуцию**
> и **охват гейтинга**, добавляем **блок free** и **env-капы**.

## Цель

Когда работу на едином диспетчере инициирует пользователь с тарифом Prime/VIP, стоимость
(`total_cost_usd`) списывается **с его профиля** — ровно на то, что наработали его задачи.
Free-пользователь диспетчер использовать не может вовсе. При исчерпании окна (5ч или 7д)
диспетчер больше не берёт работу этого профиля, пока окно не откатится.

## Ключевое правило атрибуции

**Каждая единица работы диспетчера метерится и гейтится на своего «инициатора»** (кто нажал),
а НЕ на `dispatcherUserId` (единый диспетчер) и НЕ на владельца проекта.

| Источник (`ai_usage_ledger.source`) | Инициатор = |
|---|---|
| `ai_prompt` (кнопка AI: compose/improve) | `ai_prompt_jobs.created_by` |
| `live` (воркер `claude -p` по задаче) | делегатор задачи — `task_delegations.delegator_user_id` |
| `monitoring` (AI-анализ мониторинга) | `monitoring_analysis_jobs.created_by` |
| `commit_sync` (авто-обработка коммитов, крон) | кто включил автоматизацию — сохраняем `created_by` на job'е из `SaveAutomationConfig.userId` |

Если инициатора не удаётся резолвить (legacy-NULL делегатор, отсутствует created_by) —
**fallback: не гейтим и метерим на `dispatcherUserId`** (как сейчас), чтобы не ломать поток.
Это только страховка для граничных случаев.

## Тарифы и доступ

- **Free** — **нет доступа к диспетчеру**. Любая инициация работы (AI-кнопка, «Отправить
  агенту», ручной триггер мониторинга) отклоняется с ошибкой `plan_required` (нужен Prime/VIP).
  Меняет текущее «free = безлимит»: в модели единого диспетчера у free нет своей подписки.
- **Prime / VIP** — метерятся по своим окнам; при превышении блок.

Автоматика (digest/commit-sync/monitoring), включённая free-пользователем, **не запускается**
(инициатор free → gate). Это ожидаемо: без тарифа диспетчер не работает на профиль.

## Окна и капы

- Два скользящих окна: **5ч и 7д**, блок при превышении **любого** (как построено).
- Капы берутся из `planWindowCaps(plan)` с **env-оверрайдом** (для тестов):
  `USAGE_PRIME_5H_USD`, `USAGE_PRIME_7D_USD`, `USAGE_VIP_5H_USD`, `USAGE_VIP_7D_USD`.
  Если env не задан — текущие дефолты (Prime $5.50/5ч, $22/7д; VIP $11/5ч, $44/7д).
  Пустой/невалидный env → дефолт. Позволяет поставить `USAGE_PRIME_5H_USD=0.20` и мгновенно
  поймать блок в тесте.

## Гейтинг: точки и тайминг

**Тайминг:** блокируем **старт новой** работы при превышении. Уже запущенное дорабатывает;
его стоимость дописывается в ledger и может увести юзера за лимит → тогда он заблокирован на
**следующие** запуски, пока окно не откатится. Мид-ран не убиваем.

**Точки гейта (belt-and-suspenders, вариант A+B):**
- **B — на инициации (мгновенный фидбэк):** enqueue AI-job / делегирование задачи агенту /
  ручной enqueue мониторинга проверяют бюджет инициатора → 402 `usage_blocked` (или
  `plan_required` для free) сразу в UI.
- **A — на старте прогона (авторитетный гейт):** там, где диспетчер реально начинает работу
  инициатора — 402 → диспетчер пропускает, работа ждёт отката окна:
  - `ai_prompt` / `monitoring` / `commit_sync` — на **claim** (гейт уже есть, меняем субъект
    с `dispatcherUserId` на инициатора).
  - **worker/live** — на **старте live-сессии** (`POST .../live/sessions`): сервер резолвит
    делегатора задачи и проверяет его бюджет; 402 → диспетчер не стартует, задача остаётся
    в очереди. Это новый гейт (сейчас воркер не гейтится вообще).

## Изменения по слоям

### Сервер (`server/`)

- **usage/RecordUsage:** вход получает `billedUserId` (инициатор) вместо жёсткого
  `dispatcherUserId`. Все completion-хендлеры передают инициатора.
- **usage/CheckBudget + assertBudgetAllowed:** вызвать с `billedUserId` инициатора.
- **usage/Plan.planWindowCaps:** читать env-оверрайды (через инъекцию конфига, не
  `process.env` в домене — прокинуть caps-config из composition root).
- **ai_prompt:** `CompleteAiPromptJob` → record на `job.createdBy`; гейт бюджета на `createdBy`
  в `ClaimAiPromptJob` и (новое, вариант B) в `EnqueueAiPromptJob`.
- **live/worker:** `LiveService.startSession` — резолв делегатора задачи + `assertBudgetAllowed`;
  `LiveService.finishSession` → record на делегатора (проброс `billedUserId`, вычисленного на
  старте и сохранённого на сессии, чтобы finish не перерезолвил иначе).
  Нужно донести делегатора до сессии: резолв в `startSession` из `task_delegations`.
- **monitoring:** claim/complete → субъект `job.created_by`; (опц.) гейт на enqueue.
- **commit_sync:** на job добавить `created_by` (мигр.), проставлять из включателя автоматизации
  (`SaveAutomationConfig.userId` → сохранять как enabler на конфиге, использовать при enqueue
  крон-job'а); claim/complete → субъект `created_by`.
- **Free-блок:** новый guard (напр. `assertDispatcherAccess(userId)`), бросает `plan_required`
  (HTTP 402/403) для free-инициаторов на всех точках инициации.
- **errorHandler:** маппинг `plan_required` (уже есть `UsageBlockedError`→402; добавить
  `PlanRequiredError`).

### Клиент (`client/`)

## Миграции (append-only, взять след. номера — `ls db/ | sort` → max+1)

- `commit_sync_jobs.created_by` (nullable) — инициатор (включатель автоматизации).

- `live_sessions.billed_user_id` (nullable) — инициатор (делегатор), зафиксированный на старте.

## Порядок реализации (стадии)

1. **Worker (главное — пример владельца):** `live_sessions.billed_user_id`; резолв делегатора в
   `startSession` + гейт; record на делегатора в `finishSession`. Free-блок делегирования.
2. **AI-кнопка:** атрибуция на `createdBy` (claim+complete), гейт на enqueue+claim, free-блок
   enqueue. (Опц.) правка `dispatch.ps1` для costUsd.
3. **monitoring + commit_sync:** субъект = инициатор; `commit_sync_jobs.created_by` + enabler.
4. **Env-капы** в `planWindowCaps`.
5. **Клиент:** гашение кнопок для free + апселл; обработка `plan_required`.
6. **Диспетчер:** проверить/укрепить скип на 402.

Каждая стадия: `npm run typecheck` + `lint` + `cd server && npx tsc --noEmit` зелёные, затем
коммит + пуш в `main` осмысленной порцией (автодеплой — не пушить ломающее).

## Критерии приёмки

- Prime-юзер отдаёт задачу воркеру → стоимость капает **в его** профиль (5ч/7д), видно в
  usage-баннере/диалоге/профиле. VIP — аналогично, на свой профиль.
- Free-юзер не может отдать воркеру / нажать AI — видит апселл Prime/VIP.
- При 100% окна инициатора диспетчер не берёт его новые задачи; после отката окна — снова берёт.
- Стоимость считается от `total_cost_usd`. Идемпотентность ledger сохранена (UNIQUE source+ref).
- Проверки зелёные; миграции MariaDB-совместимы, append-only.

## Риски / координация

- Фичу usage вёл параллельный сессион — правим её файлы; держим диффы точечными, работаем на
  свежем `main`, пушим малыми порциями.
